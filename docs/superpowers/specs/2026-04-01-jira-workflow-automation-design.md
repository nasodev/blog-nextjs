# 블로그 글 설계: 지라 기반 개발 자동화 워크플로우

## 메타 정보

- **슬러그**: `jira-workflow-automation-20260401-v01`
- **제목**: "워크플로우 엔지니어링 — AI 코딩 에이전트를 실무에 투입하는 방법"
- **설명**: "지라 이슈 키 하나로 스펙 작성부터 코드 개발, 5단계 검증까지 자동 완료하는 워크플로우를 설계한 과정. 프롬프트 엔지니어링의 다음 단계를 제시한다."
- **태그**: ai, claude-code, automation, jira, workflow
- **타겟 독자**: AI 코딩 에이전트를 이미 사용 중이고, 조직의 개발 프로세스에 통합하려는 시니어 개발자/테크리드
- **예상 분량**: 12,000자 이상

## 핵심 테제

프롬프트 엔지니어링의 다음 단계는 **워크플로우 엔지니어링**이다. AI 에이전트에게 "무엇을 만들어"가 아니라 "어떤 순서로, 어떤 품질 기준으로, 어떤 입출력 체계 안에서 일할 것인가"를 설계하는 것이 핵심이다. 이 글에서는 지라 이슈 키 하나로 11단계 자동화 파이프라인을 실행하는 워크플로우를 설계한 4대 원칙과 구체적 구현 방법을 공유한다.

## 접근법

**원칙 → 사례 → 구현** 구조. 워크플로우 엔지니어링이라는 개념을 먼저 정립하고, 핵심 설계 원칙 4개를 제시한 뒤, 각 원칙이 파이프라인의 어느 단계에서 어떻게 구현되는지 매핑한다.

## 도입부 스타일

**결과 선행형** — 지라 이슈 키 하나를 입력하면 벌어지는 전체 과정을 먼저 보여주고, "이 워크플로우를 설계한 과정을 공유한다"로 글의 범위를 선언.

## 11단계 도출 근거

원본 시스템은 14단계로 구성되어 있으나, 회사 특화 단계 3개를 제외하여 범용 11단계로 재구성:
- 제외: 사내 실행 추적 시스템 연동 (각 단계에서 메타데이터 전송), Google Drive 업로드, NotebookLM 인포그래픽 생성
- 이 3개는 특정 조직의 인프라에 의존하며, 워크플로우의 핵심 파이프라인(분석→생성→검증)과 무관

## 워크플로우 엔지니어링 vs 하네스 엔지니어링

이 글에서 제시하는 "워크플로우 엔지니어링"은 하네스 엔지니어링의 하위 영역이다.
- **하네스 엔지니어링**: 에이전트 바깥의 모든 것을 설계 — CI, 린터, 테스트, 관측성, AGENTS.md 등
- **워크플로우 엔지니어링**: 하네스의 구성요소 중 "에이전트의 작업 순서와 단계별 입출력 체계"를 설계하는 것
- 하네스가 "운영체제"라면, 워크플로우는 그 안에서 돌아가는 "파이프라인 스케줄러"
- 배경 섹션(2-3)에서 이 관계를 명시적으로 설명하여, 하네스 엔지니어링 글을 읽은 독자의 혼동을 방지

## 글 구조

### 1. 도입 (비중 10%)

- 지라 이슈 키 하나를 입력하면 벌어지는 일을 한 문단으로 묘사
- 11단계 파이프라인 전체 흐름을 ASCII 다이어그램으로 시각화
- "이 글에서는 이 워크플로우를 설계한 원칙과 구현 과정을 공유한다"로 범위 선언
- 톤: 담담하게 사실을 나열. 과장 없이.

### 2. 배경 — 왜 워크플로우 엔지니어링인가 (비중 15%)

**3가지 논점:**

#### 2-1. 컨텍스트 병목
- AI 에이전트의 코드 생성 능력은 이미 충분하다. 병목은 "무엇을 만들어야 하는지"를 정확히 전달하는 것
- 지라 이슈 하나에도 본문/댓글/첨부파일/연결 이슈/히스토리가 산재
- 프롬프트를 아무리 정교하게 써도, 입력 데이터가 불완전하면 출력도 불완전

#### 2-2. 일회성 생성의 한계
- "코드 만들어줘" 한 번의 요청으로는 실무 수준의 결과를 얻기 어려움
- 스펙 없이 코드를 생성하면 요구사항 누락, 스펙 없이 테스트를 생성하면 커버리지 부족
- 인간 개발자도 분석→설계→구현→검증 단계를 거치는데, AI에게는 "한 번에 다 해"라고 요청

#### 2-3. 워크플로우 엔지니어링 정의
- 프롬프트 엔지니어링: "AI에게 무엇을 어떻게 말할 것인가"
- 워크플로우 엔지니어링: "AI가 어떤 순서로, 어떤 품질 기준으로, 어떤 입출력 체계 안에서 일할 것인가"
- 단일 프롬프트가 아니라, 단계별 파이프라인을 설계하는 것
- **하네스 엔지니어링과의 관계**: 워크플로우 엔지니어링은 하네스의 구성요소 중 하나. 하네스가 운영체제라면 워크플로우는 파이프라인 스케줄러. 이전 글(하네스 엔지니어링)에서 다룬 개념의 구체적 실천 방법론.

### 3. 4대 설계 원칙 (비중 20%)

#### 원칙 1: 컨텍스트 자동화
> 에이전트가 스스로 필요한 맥락을 수집하게 하라

- 인간이 컨텍스트를 정리해서 넘기면 정보 손실 + 병목
- 지라 API로 이슈 전체 데이터 자동 수집, 소스 코드와 DB 스키마도 직접 분석
- 수집 범위를 넓게 잡되, 구조화된 포맷으로 정제
- 이 원칙은 명시적 수집 단계뿐 아니라, 각 단계에서 에이전트가 코드베이스를 직접 읽고 분석하는 행위에도 적용

#### 원칙 2: 단계적 정제
> 각 단계의 출력이 다음 단계의 입력이 되는 파이프라인을 만들어라

- 원시 데이터 → 스펙 → 계획 → 코드로 점진적 정제
- 각 단계의 출력물이 명확한 포맷(마크다운, JSON) → 다음 단계의 입력 품질 보장
- 중간 산출물이 파일로 남아 인간이 개입할 수 있는 리뷰 포인트 생성
- 모든 산출물이 파일 시스템에 기록되므로, 의사결정 과정을 사후 추적 가능

#### 원칙 3: 자기 검증 루프
> 생성한 결과를 스스로 교차 검증하는 단계를 반드시 넣어라

- AI는 자신감 있게 틀린다. 생성과 검증을 같은 단계에서 하면 같은 편향으로 통과
- 생성 단계와 검증 단계 분리: 스펙→리뷰, 계획→리뷰, 코드→검증
- 검증 실패 시 재생성 루프 (최대 N회 → 실패 시 인간 에스컬레이션)

#### 원칙 4: 환경 격리
> 자동화 실행 환경을 원본과 격리하라

- Git worktree로 이슈별 독립 환경 — 원본 저장소를 절대 오염시키지 않음
- 실패해도 worktree 삭제로 원상복구, 성공하면 PR로 병합
- 이슈별 독립 브랜치/디렉토리/의존성 — 여러 이슈를 동시에 작업해도 충돌 없음

#### 원칙 ↔ 파이프라인 매핑 테이블

●: 해당 단계의 핵심 원칙, ○: 부수적으로 적용되는 원칙

| 파이프라인 단계 | 원칙 1: 컨텍스트 자동화 | 원칙 2: 단계적 정제 | 원칙 3: 자기 검증 | 원칙 4: 환경 격리 |
|---|:---:|:---:|:---:|:---:|
| 이슈 컨텍스트 수집 | ● | ○ | | |
| 스펙 생성 | ● | ● | | |
| 스펙 리뷰 | | | ● | |
| BDD 시나리오 | | ● | | |
| 개발 계획 | ○ | ● | | |

### 4. 파이프라인 설계 상세 (비중 25%)

11단계를 6개 하위 섹션으로 묶어 설명. 각 하위 섹션마다 **설계 의도 → 구체적 동작 → 입출력 포맷 → 설계 시 고려사항**.

#### 4-1. 이슈 컨텍스트 수집 (원칙 1)
- 지라 REST API로 수집하는 데이터 범위: 이슈 본문, 댓글(작성자/시각), 첨부파일(텍스트 전문 추출), 연결 이슈(1-depth), 변경 이력
- 외부 URL 자동 크롤링
- 출력: 구조화된 JSON(기계용) + 마크다운 리포트(사람용) 이중 포맷
- "넓게 수집하고, 다음 단계에서 정제한다"

#### 4-2. 스펙 생성 + 스펙 리뷰 (원칙 1,2,3)
- 수집 컨텍스트 + 소스 코드 분석 + DB 스키마 분석 → 개발 명세서 합성
- 이슈 유형 자동 분류(feature/bug/enhancement/refactor) → TDD/BDD/E2E 적용 여부 결정
- 리뷰: 팩트 체크(파일 경로/테이블 존재 확인), 범위 검증, 요구사항 완전성
- 판정: APPROVED / REVISE / REGENERATE (최대 2회)

#### 4-3. BDD + 개발 계획 + 계획 리뷰 (원칙 2,3)
- BDD: 스펙에서 Given-When-Then 시나리오 자동 도출 (feature/enhancement만)
- 개발 계획: 파일 단위 태스크 분해, 코드 스니펫 포함
- 기준: "외부 개발자가 계획서만 보고 구현할 수 있는 수준"
- 계획 리뷰: 스펙 커버리지 + 파일 경로 실존 검증

#### 4-4. 격리 환경 + 자동 개발 (원칙 2,4)
- Git worktree로 이슈별 독립 브랜치/디렉토리
- 의존성 설치, 환경변수 배포까지 자동화
- TDD: RED→GREEN 사이클, 태스크 단위 커밋
- 커밋 메시지에 이슈 키 포함

#### 4-5. 5단계 검증 + E2E 테스트 (원칙 3)
- 기능 → 타입 체크 → 보안(OWASP Top 10) → 성능 → 코드 품질
- 5단계는 범용적 검증 체계: 어떤 프로젝트에든 적용 가능한 범주
- 실패 시 수정 후 전체 재검증 (최대 5회)
- 5회 실패 → 인간 에스컬레이션
- E2E: 서버 기동 → 브라우저 자동화 → 에러 체크 → 스크린샷

#### 4-6. 보고서 생성 (원칙 2)
- 모든 단계 산출물 종합 완료 보고서
- HTML 시각화: 인라인 이미지, 스크린샷, 검증 결과를 단일 파일로
- 파이프라인의 마지막 정제 단계: 개별 산출물 → 통합 보고서

### 4-7. 에러 처리 전략

파이프라인 전체에 걸쳐 적용되는 통합 에러 처리 철학:

**에러 분류:**
| 유형 | 예시 | 대응 |
|------|------|------|
| 외부 시스템 장애 | 지라 API 불가, DB 연결 실패 | 즉시 중단, 인간 에스컬레이션 |
| 품질 미달 | 스펙 리뷰 REGENERATE, 검증 실패 | 재생성 루프 (단계별 최대 횟수 설정) |
| 실행 불가 | 파일 경로 불일치, 테스트 프레임워크 미설치 | 즉시 중단, 구체적 에러 메시지와 함께 에스컬레이션 |
| 비핵심 실패 | 보고서 시각화 실패 | 경고 로그 후 계속 진행 |

**핵심 원칙:**
- "추측하지 않는다" — 불확실한 상황에서 임의로 진행하지 않고 즉시 멈춤
- 재시도 횟수 상한을 반드시 설정 (무한 루프 방지)
- 모든 에러는 보고서에 기록되어 사후 분석 가능

### 5. 구현 가이드 — Claude Code 스킬 시스템 (비중 25%)

#### 5-1. 스킬 아키텍처 개요
- Claude Code 스킬: `/슬래시 커맨드`로 호출 가능한 마크다운 기반 지시서
- 디렉토리 구조: `.claude/skills/{스킬명}/SKILL.md`
- 스킬 간 데이터 전달: 파일 시스템 기반 (이전 스킬의 출력 → 다음 스킬의 입력)
- 각 스킬이 독립적인 역할 + 명확한 입출력 인터페이스

**SKILL.md 파일 구조:**
```markdown
---
name: skill-name
description: 한 줄 설명
---

## 입력
- 필요한 파일/데이터 명시

## 실행 절차
1. 단계별 지시사항
2. 조건부 분기 로직
3. 출력 포맷 강제

## 출력
- 생성할 파일 경로와 포맷

## 에러 처리
- 블로커 조건과 대응 방법
```

핵심: SKILL.md는 "코드"가 아니라 "지시서"다. 에이전트가 읽고 따르는 마크다운 문서이며, 실행 절차, 출력 포맷, 에러 조건을 자연어로 정의한다.

#### 5-2. 오케스트레이터 패턴
- N개 스킬을 순차 호출하는 마스터 스킬 설계
- 조건부 실행 로직: 이슈 유형에 따라 BDD/E2E 스킵
- 에러 처리: 블로커 발생 시 중단 + 에스컬레이션 규칙
- 오케스트레이터 SKILL.md 실행 흐름 예시를 의사코드 형태로 제시:
  ```
  Step 1: /init → JIRA_KEY, REPORTS_DIR 획득
  Step 2: /collect → context.json 생성
  Step 3: /spec-generate → spec.md 생성
  Step 4: /spec-review → APPROVED? → 아니면 Step 3 재실행 (max 2)
  Step 5: if spec.BDD == true → /bdd → bdd-scenarios.md
  Step 6: /plan → dev-plan.md
  ...
  ```

#### 5-3. 지라 API 연동 스킬 구현
- 지라 REST API v2 인증(HTTP Basic Auth) 및 호출 방법
- 수집 스크립트 구조: Python 스크립트를 스킬에서 Bash로 실행
- context.json 최상위 스키마:
  ```json
  {
    "issue": { "key": "", "summary": "", "description": "", "type": "" },
    "comments": [{ "author": "", "body": "", "created": "" }],
    "attachments": [{ "filename": "", "content": "" }],
    "linked_issues": [{ "key": "", "summary": "", "relation": "" }],
    "changelog": [{ "field": "", "from": "", "to": "", "date": "" }]
  }
  ```
- 외부 의존성 없이 urllib만으로 구현하는 접근 (배포 환경 제약 최소화)

#### 5-4. 스펙 생성 스킬 구현
- 입력(context.json + 코드 분석)을 프롬프트로 구성하는 방법
- 출력 포맷 강제: 마크다운 템플릿 기반 스펙 생성
- 이슈 분류 로직과 플래그(TDD/BDD/E2E) 결정 흐름

#### 5-5. 검증 스킬 구현
- 5단계 검증 각각의 체크리스트를 스킬 내에 정의
- 재시도 루프: 실패 → 수정 → 재검증 반복 로직
- 최대 재시도 횟수와 에스컬레이션 임계값

#### 5-6. 나머지 스킬 구현 안내
- 5-3~5-5에서 다룬 3개 스킬(수집, 스펙 생성, 검증)이 파이프라인의 대표 패턴
- 나머지 스킬(BDD, 계획, 리뷰, worktree, 개발, 보고서)도 동일한 구조를 따름:
  - SKILL.md에 입력/절차/출력/에러 정의
  - 이전 단계의 출력 파일을 읽어 다음 단계 실행
  - 필요시 외부 도구(bash 스크립트, Python 등) 호출
- 독자가 자기 프로젝트에 맞게 변형할 포인트: 검증 체크리스트 항목, BDD 적용 조건, 커밋 메시지 포맷 등

### 5-7. 비용과 토큰 관리

- 11단계 파이프라인 1회 실행 시 다수의 LLM 호출 발생
- 비용 최적화 전략:
  - 컨텍스트 수집 단계는 API 호출 + 파일 I/O 중심 → LLM 토큰 소비 최소
  - 검증 단계 중 타입 체크, 린트는 도구 실행으로 대체 (LLM 불필요)
  - 재시도 횟수 상한 설정으로 비용 폭주 방지
  - 조건부 실행(BDD/E2E 스킵)으로 불필요한 단계 생략
- 실측 참고: 구체적 비용은 프로젝트 규모와 이슈 복잡도에 따라 크게 다름. 독자가 자기 환경에서 측정할 것을 권장

### 5-8. 성공 지표

워크플로우 자동화가 잘 작동하는지 판단하는 지표:
- **파이프라인 완주율**: 11단계를 인간 개입 없이 완료한 비율
- **검증 1회 통과율**: 5단계 검증을 재시도 없이 통과한 비율
- **인간 에스컬레이션 빈도**: 어느 단계에서 주로 멈추는가 → 해당 단계 스킬 개선 신호
- **PR 병합률**: 자동 생성된 코드가 실제 리뷰를 통과해 병합된 비율
- 이 지표들을 보고서에 누적 기록하면, 워크플로우 개선의 근거 데이터가 됨

### 6. 마무리 (비중 5%)

- "프롬프트 엔지니어링 → 워크플로우 엔지니어링" 테제 재확인
- AI 코딩 에이전트의 역할이 "코드 생성기"에서 "개발 프로세스 실행기"로 전환
- 워크플로우를 설계하는 것은 "좋은 개발 프로세스가 무엇인가"를 정의하는 것
- AI 시대에도 소프트웨어 엔지니어링의 본질은 변하지 않는다
- 짧게, 담백하게 마무리

## 작성 스타일 가이드

- **언어**: 한국어, 기술 용어는 영어 혼용
- **톤**: 전문적, 직접적, 의견 있는 서술. 과장/감탄 지양
- **구조**: 계층적 헤더(##, ###, ####), 테이블, 코드 블록 적극 활용

- **구체적 사례**: 추상적 설명보다 실제 동작 예시 우선
- **참고**: 기존 블로그 글(harness-engineer, sonar-claude-workflow)의 톤과 구조를 따름
